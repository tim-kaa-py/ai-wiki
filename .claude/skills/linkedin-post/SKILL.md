# LinkedIn Post Skill

Produces a LinkedIn-ready post from an existing `summaries/<slug>.md` file. The post has two distinct parts that must remain visually and textually separated:

1. **Personal note** — written by the user, lightly polished, clearly labeled as human-authored.
2. **Claude Code generated summary** — 3 curated Key Takeaways, clearly labeled as AI-generated.

## Why this separation matters

The user positions themselves as a Claude Code expert who shares learnings transparently. Some of their audience is skeptical of AI-generated content. The personal note (when present) must read as *unmistakably human* — that is the trust anchor. The AI portion must be clearly labeled so nobody feels misled, and the "generated by Claude Code" wording doubles as positioning (the user knows the tool, uses it openly, and invites others to join the journey).

**Important framing:** the human contribution is *curation*, not *authorship*. The user reads the source, chooses which takeaways they found most interesting, and lets Claude Code generate the written summary from those picks. The skill must make this clear in the post itself — the takeaways are user-selected ("3 takeaways I picked from this source") but the prose around them is AI-generated ("summary generated by Claude Code"). This honors both sides: the user's editorial taste is real, and the wording is transparently machine-produced.

This is why the skill treats the personal note with extreme editorial restraint — typos and obvious mistakes only, never restructuring, never "improving" voice or tone.

## Inputs

- **Required:** path to a summary file, e.g. `summaries/2026-04-15_latent-space_notion-token-town-mcp-clis-software-factory.md`. The user will usually paste this or give the slug.
- **Required (interactive):** the user's personal note — a few sentences in their own voice explaining why they're sharing this.
- **Optional:** language override. Default is English. If the user says "in German" or "auf Deutsch" when invoking, use German for the entire post including labels and hashtags.

## Workflow

### 1. Resolve the summary

If the user gave a full path, use it. If they gave only a slug, resolve to `summaries/<slug>.md`. Read the file and extract:
- Title
- Source URL
- Tags (from frontmatter)
- TL;DR
- Key Takeaways (full list)

If the summary doesn't exist, stop and tell the user — don't try to ingest from a URL. This skill consumes existing wiki summaries.

### 1a. Find the canonical source URL

The summary's `url` field may point at an aggregator (Apple Podcasts, Spotify, YouTube rehost) rather than the creator's own page. The user prefers linking to the **original source** when one exists, to give proper credit to the creator.

Check the URL against these patterns:

| URL host | Likely aggregator? | Action |
|----------|-------------------|--------|
| `podcasts.apple.com` | Yes — aggregator | Try to find the podcast's own site |
| `open.spotify.com` | Yes — aggregator | Try to find the podcast's own site |
| `youtube.com` / `youtu.be` | Depends — YouTube is original for many creators | Keep unless the episode exists on the creator's own site too |
| `<creator-site>.com` / `<creator-site>.substack.com` / personal domain | No — this IS canonical | Use as-is |
| arxiv.org, github.com, creator blog | No — canonical | Use as-is |

**If aggregator:** do a WebSearch for `<podcast name> <episode title>` and look for a result on the podcast's own website (e.g. `latent.space`, `dwarkeshpatel.com`, `lennysnewsletter.com`). If found, use that URL. If not found after one search, fall back to the aggregator URL.

Tell the user which URL you're using and why, so they can override if needed.

### 2. Ask for the personal note (optional)

Ask the user one question: "Personal note for this post? Write it in your own voice — I'll only fix typos. Or say 'skip' to post just the AI summary."

Wait for their response. If they reply "skip", "no", "none", "just the summary", or similar, proceed without a personal note and omit Step 3 entirely. Otherwise accept whatever they write (one line or paragraph) and proceed with Step 3.

**Why optional:** the user does not have time to write a sharp personal angle for every post. Forcing one leads to generic "fascinating to see how [company] pushes the boundaries" filler that weakens the post and undermines the trust anchor. Better to skip the personal note entirely than ship lukewarm filler — the `🤖 Claude Code generated` label alone carries the transparency and positioning.

### 3. Polish the personal note — minimally

The goal is to make the note *publishable*, not *improved*. Allowed changes:

- Obvious typos (e.g. "teh" → "the", missing space after period)
- Missing capital at the start of a sentence
- Missing punctuation at the end of a sentence
- Obvious auto-correct failures (e.g. "their" vs "there" when context makes the intent unambiguous)

**Not allowed:**
- Rephrasing for clarity
- Reordering sentences
- Combining or splitting sentences
- Changing word choice ("big" → "significant")
- Adding transitions or connective tissue
- Adding hashtags, emojis, or formatting the user didn't use
- "Polishing" grammar that is stylistically intentional (e.g. fragments, em-dashes, casual tone)

If in doubt, leave it as the user wrote it. A slightly rough human voice beats a smoothed-out AI voice for this purpose.

Show the user a diff or before/after only if you changed more than one or two characters. If it was a single obvious typo fix, just proceed.

### 4. Curate 3 Key Takeaways

From the summary's Key Takeaways list, pick the **3 most insightful or actionable** for a general LinkedIn audience. Criteria:

- Prefer takeaways with concrete "How to apply" guidance
- Prefer non-obvious insights over restated common knowledge
- Prefer claims that stand alone without requiring the full video context
- Avoid deeply technical takeaways that only make sense to specialists (unless the whole post targets them — check tags)

Show the user the 3 you picked in a numbered list. Ask: "These look good, or swap any? (reply 'ok' or e.g. '2, 5, 7' to pick different ones)." Wait for confirmation.

### 5. Condense each takeaway

For the post, each takeaway should be **1-2 short lines**, not a paragraph. Strip the "How to apply:" sublines into the takeaway itself where possible, or drop them if they add length without clarity. Keep the creator's framing — don't editorialize.

### 6. Draft the post

**Variant A — with personal note:**

```
<Hook line — 1 short sentence, often a provocative takeaway or question. This is what appears before LinkedIn's "see more" cutoff, so it must earn the click.>

<Personal note — the polished version from Step 3, as-is. Preserve the user's line breaks.>

— Written by me ✍️

———

🤖 3 takeaways I picked from this source — summary generated by Claude Code:

▸ <Takeaway 1 — 1-2 lines>

▸ <Takeaway 2 — 1-2 lines>

▸ <Takeaway 3 — 1-2 lines>

———

I share this to invite you on my knowledge journey.

Source: <canonical URL from Step 1a>

<3-5 hashtags derived from the summary's tags + #ClaudeCode>
```

**Variant B — without personal note (default when user skips):**

```
<Hook line — 1 short sentence distilling the most striking takeaway, phrased to make someone stop scrolling. This must earn the click past LinkedIn's "see more" cutoff.>

———

🤖 3 takeaways I picked from "<title>" (<channel>) — summary generated by Claude Code:

▸ <Takeaway 1 — 1-2 lines>

▸ <Takeaway 2 — 1-2 lines>

▸ <Takeaway 3 — 1-2 lines>

———

I share this to invite you on my knowledge journey.

Source: <canonical URL from Step 1a>

<3-5 hashtags derived from the summary's tags + #ClaudeCode>
```

**Notes on both structures:**

- The hook line is always present. In Variant B it does double duty (attention + signals what the post is about) so it must be sharper.
- The "— Written by me ✍️" line is non-negotiable **when a personal note is present** — it's the human-authorship signal.
- The "🤖 Claude Code generated summary" line is non-negotiable in both variants. It's the AI-transparency signal *and* the positioning signal.
- The `———` dividers (em-dashes, not hyphens) create visual separation that survives LinkedIn's formatting stripping.
- The closing invitation ("join my knowledge journey") is the user's stated mission — keep it.
- **The source link goes in the post body**, not in a first comment. Reach optimization is deprioritized in favor of reader convenience and proper attribution to the original creator. Prefer the canonical source (creator's own site) over aggregators (Apple Podcasts, Spotify) — see Step 1a.

### 7. (No separate first comment)

The source link lives in the post body (see structure above). No first comment is produced.

### 8. Length check

Target the main post at **1200-1800 characters**. If under 1200, the post feels thin — consider a slightly longer hook or a fourth short line. If over 1800, shorten takeaways (users scroll past walls of text on LinkedIn). Don't cut the labels or dividers — those are the trust scaffolding.

### 9. Hashtags

Derive 3-5 hashtags from the summary's `tags` frontmatter. Convert to LinkedIn format: `ai-agents` → `#AIAgents`, `claude-code` → `#ClaudeCode`, `prompt-engineering` → `#PromptEngineering`. Always include `#ClaudeCode` (user's positioning). Drop tags that don't translate well (e.g. internal taxonomy tags like `opinion`, `how-to`).

### 10. Language override

If the user specified German: translate labels too.
- "— Written by me ✍️" → "— Von mir geschrieben ✍️"
- "🤖 3 takeaways I picked from this source — summary generated by Claude Code:" → "🤖 Von Claude Code generierte Zusammenfassung der Quelle:"
- "I share this to invite you on my knowledge journey." → "Ich teile das, um euch auf meiner Lernreise mitzunehmen."
- "Source:" → "Quelle:"

Translate the hook and takeaways into natural German (not literal). Keep hashtags in English where they are industry-standard terms (#ClaudeCode, #AIAgents), mix in German where natural.

### 11. Save the post

Save to `posts/<slug>.md`. Create the `posts/` directory if it doesn't exist. Use this frontmatter:

```yaml
---
source_summary: "summaries/<slug>.md"
source_url: "<url>"
created: "<YYYY-MM-DD>"  # today's date
language: "en"  # or "de"
status: "draft"  # user updates to "posted" manually
---
```

File body:

```markdown
# LinkedIn Post — <Title>

## Main Post

<the full post text>

## Source URL used

<canonical URL> (aggregator URL from summary was: <original URL>, if different)

## Character Count

Main post: <N> characters
```

### 12. Do NOT touch the wiki

This skill produces personal publishing artifacts, not knowledge. Do not:
- Update `index.md`
- Append to `log.md`
- Create or modify any `wiki/` page
- Add the post slug to any source/summary frontmatter

The `posts/` directory lives alongside the wiki but is separate from it.

### 13. Present to user

Show the final post and the first comment in the chat, plus the file path where it was saved, plus the character count. Offer to tweak the hook, swap takeaways, or adjust length.

## Example

**Invocation:** "Make a LinkedIn post from `summaries/2026-04-15_latent-space_notion-token-town-mcp-clis-software-factory.md`"

**Skill reads summary → asks for personal note.**

**User responds:** "i've been thinking alot about how software is becoming a factory process with agents. this episode had the best framing i've seen so far especially the part about MCP vs CLIs"

**Skill polishes:** "I've been thinking a lot about how software is becoming a factory process with agents. This episode had the best framing I've seen so far — especially the part about MCP vs CLIs."

(Changes: "alot" → "a lot", capital I, period at end, em-dash before "especially" for readability of the fragment. Nothing else.)

**Skill picks 3 takeaways, confirms with user, drafts post, saves to `posts/2026-04-15_latent-space_notion-token-town-mcp-clis-software-factory.md`.**

## Guardrails

- Never invent takeaways — only use what's in the summary.
- Never rewrite the personal note beyond typo fixes. If tempted to "improve" it, stop.
- Never omit the "🤖 Claude Code generated" label. Never omit "— Written by me ✍️" when a personal note is present.
- Always include the source URL in the post body. Prefer the canonical source (creator's own site) over aggregator URLs when a canonical one exists.
- Never modify the wiki (`index.md`, `log.md`, `wiki/`, `summaries/`, `sources/`).
- Default language is English. Only use German if the user explicitly asked for it in the current invocation.
