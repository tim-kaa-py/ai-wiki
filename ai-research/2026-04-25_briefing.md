## Claude Code Updates

### [v2.1.120 — ultrareview in CI](https://www.youtube.com/watch?v=_8cOVcgTIeU)

- **ultrareview escapes the terminal** — `/ultrareview` can now run directly from CI without an interactive Claude Code session. Point it at a branch or PR, pipe findings with `--json`, and key off the exit code (0 = complete, 1 = failure).
  - **So what:** You can now gate PRs on a deep multi-agent code review in your CI pipeline — no developer needs to be in the terminal to trigger it.

- **Windows: no more git bash requirement** — Claude Code now falls back to PowerShell automatically when git bash is absent.
  - **So what:** Windows users on teams without a standardized shell setup can run Claude Code out of the box without extra configuration.

- **Critical file descriptor bug fixed** — The `find` tool was exhausting all open file descriptors on large repos, potentially taking down the host process.
  - **So what:** Large monorepo users should update immediately; this was a silent host-stability risk.

---

### [v2.1.119 — Settings That Actually Stick](https://www.youtube.com/watch?v=rkac5fIfVz4)

- **Persistent config settings** — Theme, editor mode, and verbose flag set via `config` now write to `~/.claude/settings.json` and survive restarts. The full project → local → policy override hierarchy is respected.
  - **So what:** You can set your preferred theme and verbosity once and never re-enter them; settings now behave like a real config file rather than session-only state.

- **Multi-platform PR support** — `--from-pr` now resolves merge requests and pull requests from GitLab, Bitbucket, and GitHub Enterprise URLs, not just github.com.
  - **So what:** Teams on self-hosted or non-GitHub platforms can now use Claude Code's review features without workarounds.

---

### [v2.1.118 — Vim Visual Mode & Custom Themes](https://www.youtube.com/watch?v=hPeNrtzvCig)

- **Vim visual mode** — `v` for character-wise, `V` for line-wise visual selection. Operators, selections, and visual feedback all work as expected.
  - **So what:** Vim-native users can now select and operate on text in the terminal editor without reaching for the mouse or adapting to non-Vim patterns.

- **Named themes + full theme control** — Create and switch named themes with the `theme` command, hand-edit JSON in `~/.claude/themes/` for granular control. Plugins can ship their own themes via a `themes/` directory.
  - **So what:** Use this to unify the Claude Code interface with your terminal color scheme, or ship a team theme as part of a plugin to standardize developer environments.

---

### [v2.1.117 — Opus 4.7 Fixed, Faster Search](https://www.youtube.com/watch?v=X13MxJNLRag)

- **Opus 4.7 context window corrected** — Claude Code was computing context usage against a 200K window; Opus 4.7's actual limit is 1M tokens. Auto-compact triggers and context percentage displays are now accurate.
  - **So what:** Opus 4.7 sessions were compacting prematurely, truncating long tasks. After this fix, long agentic runs on Opus 4.7 will last significantly longer before hitting context limits.

- **Native build: glob/grep replaced by embedded BFS + ugrep** — On macOS and Linux native builds, the standalone `glob` and `grep` tools are gone, replaced by BFS and ugrep running through the bash tool directly. No separate tool round-trip.
  - **So what:** File search operations are faster on native builds; Windows and NPM installs are unchanged.

---

### [v2.1.111 — Opus 4.7 xhigh & /ultrareview](https://www.youtube.com/watch?v=pcRkn0QTED4)

- **New "xhigh" effort tier** — A gear between `high` and `max`. Run `effort` with no arguments to open an interactive slider; use arrow keys + Enter to select. Pick `xhigh` for hard tasks, drop to `high` when not needed.
  - **So what:** You get a more granular cost/quality dial — `xhigh` lets you spend more on a specific hard problem without jumping all the way to `max` for everything.

- **/ultrareview introduced** — Run a comprehensive cloud-based code review on your current branch or a specific GitHub PR number. Parallel agents analyze and critique; described as "the review you'd want from a careful senior engineer," not a linter.
  - **So what:** Use `/ultrareview` before merging significant changes to catch architectural issues and subtle bugs that a linter won't surface.
