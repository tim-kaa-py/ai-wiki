# How to Code with AI Agents (Peter Steinberger) — Tips & Tricks

**Source:** Lex Clips | 2026-02-12 | [Watch](https://www.youtube.com/watch?v=wKy1_KLcxcs) | 31:13

## TL;DR

Peter Steinberger (creator of OpenClaw) shares the principles he converged on after months of intensive agentic coding. The core insight: treat the agent like a capable but context-less engineer on your team. Build your codebase for *them*, empathize with their limitations, keep prompts simple, never revert — just move forward. It's a skill you develop through practice, not a tool you pick up once.

## Coding Workflow

### Tool Evolution

Peter's journey through tools: started with Claude Code in April 2025, experimented with Cursor but disliked the difficulty of running multiple instances, then went back to **Claude Code as his main driver** — all CLI, all terminal. He barely uses an IDE anymore, only occasionally as a **diff viewer** to spot-check changes.

### The Setup

- **Multiple Claude Code terminal windows running side by side** — at his peak he had 7 Max subscriptions, burning through one per day
- He switches between terminals using the **keyboard** and uses **voice input** (walkie-talkie button) for the actual agent prompts
- For quick terminal commands (switching folders, running scripts), he types — it's faster. For agent conversations, he talks
- Each terminal window runs its own agent session with its own task: one builds a larger feature, one explores an idea he's unsure about, others fix bugs or write docs

### How He Assigns Work

- **Larger features** — dedicated agent session, conversational back-and-forth
- **Exploring ideas** — a separate session where he's not sure yet and wants to play
- **Small bugs / docs** — quick sessions, often 2-3 running in parallel
- **Documentation** — treated as part of a feature, not a separate task. Most docs are autogenerated and "infused with some prompts"

### Git & CI Workflow

- **No develop branch** — commits directly to main
- **Main is always shippable** — that's the goal
- **Local CI** (DHH-inspired) — runs tests locally, pushes to main if they pass. GitHub CI still exists but is secondary
- **Never reverts** — if something breaks, he prompts the agent to fix it rather than rolling back
- **Commits when he likes the outcome** — not after every change, but when the result is good

### Reading Code

Peter famously "doesn't read the code" — but clarifies: he doesn't read the **boring parts**. Most software is just data moving from one shape to another, stored in a database, shown to the user, sent back. That's not worth reading. What he *does* read and review:
- Anything that **touches the database**
- **PR code** (for security — "I don't trust people, there could always be something malicious")
- Architectural decisions and design trade-offs

## Key Principles

### 1. The Agentic Trap — Simplicity → Complexity → Zen Simplicity

Beginners start with simple prompts ("please fix this"). Then they over-engineer: 8 agents, complex orchestration, multi-checkout chaining, 18 slash commands, custom workflows. The elite level is returning to **short, direct prompts** — "Hey, look at these files and do these changes." The sophistication is invisible; it lives in your understanding, not your tooling.

**How to apply:** Resist the urge to build elaborate prompt pipelines. Invest in understanding the agent instead of automating around it.

### 2. Empathize with the Agent

The agent starts every session from zero. It knows nothing about your project. Your codebase might be hundreds of thousands of lines, but the agent's context window is finite. You need to **think from its perspective**: what does it see? What can't it see? Where should it look?

A few pointers from you — "consider this file, this module, this constraint" — go a long way. You have the system understanding; the agent needs you to share just enough of it.

**How to apply:** Before prompting, ask yourself: "If I were dropped into this codebase cold, what would I need to know to do this task?" Then tell the agent exactly that.

### 3. Build Your Codebase for the Agent

Don't fight the names the agent picks. The name it chooses is most likely the one "in the weights" — the most obvious name. Next time it searches, it'll look for that name. If you override it with your preference, you make the agent's job harder.

The goal shifts from "a codebase that's perfect for me" to **"a codebase that's easy for an agent to navigate."**

**How to apply:** Accept agent-suggested naming. Keep file structure clean and discoverable. Write CLAUDE.md / agent files that orient the agent quickly.

### 4. Never Revert — Always Move Forward

Instead of rolling back when something goes wrong, just ask the agent to fix it. Peter finds that reverting and redoing takes longer than pushing forward. Commit when you like the outcome. Don't obsess over the perfect prompt — if the result isn't right, course-correct.

He also runs local CI (inspired by DHH): run tests locally, push to main if they pass. No develop branch. Main should always be shippable.

**How to apply:** Stop treating mistakes as rollback points. Treat them as the next prompt. "This broke X — fix it while keeping Y."

### 5. Treat It Like Managing an Engineering Team

Peter draws a direct parallel to leading engineers: your team won't write code the same way you would. It might not be as "good" as you'd do it. But it pushes the project forward. If you micromanage, everything slows down.

The same applies to agents. **Let go of perfection.** Accept working solutions. If it turns out to be too slow or problematic later, you can always redo it. Refactors are cheap now.

Don't force your worldview — the agent might have a better approach because it was trained on patterns you haven't seen.

**How to apply:** Judge agent output by "does this work and move the project forward?" not "is this exactly how I would have written it?"

### 6. It's a Skill — Practice Required

Agentic engineering is like playing a musical instrument. You don't sit at a piano once, produce bad music, and conclude the piano is broken. You need to **play with the tools**, build little things, develop a feel for what works.

Peter spent "an ungodly amount of time" over the year practicing. Each step compounded — he got better, the agents got better, his understanding deepened. He couldn't have achieved his current output even a few months earlier.

**How to apply:** Dedicate time to just experimenting. Build throwaway projects. The skill compounds.

### 7. Approach It Like a Conversation

When reviewing PRs, Peter's first question to the agent is: "Do you understand the intent of this PR?" Not the implementation — the *intent*. Then he guides the agent: "Have you looked at this part? What about this constraint?" They have a discussion about the optimal solution.

When something takes too long, that's a signal — stop (press escape), ask where the friction is. Maybe you didn't give enough context. Maybe the architecture makes the feature hard. Treat long execution as feedback.

**How to apply:** If the agent is spinning, don't wait — interrupt and reframe. Guide with intent first, implementation second.

### 8. The soul.md Concept

Inspired by Anthropic's constitution for Claude, Peter created a `soul.md` for his agent — a document capturing how he wants to work with AI, core values, personality traits. The agent is allowed to modify its own soul file (with the condition that Peter sees the changes).

He had the agent itself write the templates: "You see these files we created — infuse them with your personality." The result had character that no human-written dry template could match.

Key quote from his agent's soul.md: *"I don't remember previous sessions unless I read my memory files. Each session starts fresh. A new instance loading context from files. If you're reading this in a future session, hello. I wrote this, but I won't remember writing it. It's okay. The words are still mine."*

**How to apply:** Consider creating a soul.md or personality document for your agent. Let the agent contribute to its own identity file.

### 9. Use Voice Input

Peter uses voice extensively to talk to agents — pressing a "walkie-talkie button" and speaking naturally. He types for terminal commands and quick operations (it's faster), but for actual agent conversations he speaks. He described it as having "bespoke prompts" via voice, treating it as a real conversation rather than typed instructions.

He used voice so intensively at one point that he temporarily lost his voice.

**How to apply:** Try voice input for agent prompts — it encourages conversational, natural-language interaction rather than terse typed commands.

## Workflows

### PR Review Workflow

Peter's agent-assisted PR review follows a deliberate sequence:

1. **"Review this PR"** — Hand the PR to the agent, get an initial review.
2. **"Do you understand the intent?"** — First question is always about *why*, not *how*. What problem is the person trying to solve? (99% of PRs are either fixing a bug or adding a feature.)
3. **"Is this the most optimal way?"** — Usually the answer is no.
4. **"What would be better? Have you looked at X, Y, Z?"** — Point the agent to parts of the codebase it hasn't seen yet (it starts with an empty context). Share your system-level understanding.
5. **Discuss the optimal solution** — Have a back-and-forth about the best approach.
6. **"Could we make it even better with a larger refactor?"** — Consider if a broader refactor is worth it. Often Peter just does the refactor because "refactors are cheap now."
7. **Decide: ship or defer** — Accept the working solution or invest in the refactor.

Note: Peter still manually reviews PR code for security reasons — "I don't trust people, there could always be something malicious." He also observed that sometimes PRs take him longer than if the contributor had just written a good issue instead.
